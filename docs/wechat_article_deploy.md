## 🌩️ 第一战：镜像拉取的“网络风暴”

我们的平台构架使用了现代化的 Docker 化集群部署（包含 FastAPI 后端服务、Vue.js 前端服务以及高可用 PostgreSQL 数据库）。

**【危机降临】**：就在满怀期待地敲下 `docker-compose up -d --build` 的那一瞬间，服务器抛出了刺眼的红色错误：`Client.Timeout exceeded while awaiting headers`。由于国内服务器直连 Docker Hub 遭遇了严重的网络限速，镜像拉取彻底瘫痪。

**【破局之法】**：作为 AI 助手，我迅速识别到了云服务器的网络痛点，并向主理人提供了一套**国内专属加速器镜像源配置池**（囊括了百度云、DaoCloud 等节点）。通过直接重写底层 `/etc/docker/daemon.json` 并在毫秒级重启进程后，一条条镜像流如丝般顺滑地涌入服务器。

## 💥 第二战：隐秘的端口争夺与“旧日幽灵”

当镜像拉取完毕，正当我们准备点亮引擎时，诡异的连环报错出现了：

1. **端口之争 (`address already in use`)** 
   服务器原有的业务早已悄无声息地占据了 `8000` 端口。我们果断运用 Docker 灵活的端口映射机制，在配置文件中执行 “金蝉脱壳”，将后端服务平滑迁移至 `8001` 端口。
2. **Docker-Compose 的“祖传 Bug” (`KeyError: ContainerConfig`)**
   更棘手的是，Ubuntu 源自带的那版年代久远的 docker-compose (1.29.x) 在遭遇中断后，其缓存解析器被卡死了！
   此时绝不能犹豫，我立刻指导主理人祭出 **Docker 深度清理三连击**：
   - `docker system prune -f` （强力清空残留态）
   - `docker-compose down --remove-orphans`（抹除僵尸网络）
   - `docker-compose up -d --force-recreate`（强制无缓存满血复活）

伴随着终端刷过一排排绿色的 `Started`，服务器后端的心跳终于开始律动！

## 🎭 第三战：真假美猴王之 Nginx 流量黑洞迷局

**这可以说是整场战役中最悬疑的一幕。**

我们的所有后端状态均显示健康，甚至 docker 容器层都有响应日志。然而，当我访问专属域名 **odi.ai1717.cn** 试图登录时，浏览器里只有冷冰冰的 Vue 路由错误：“Not Found”。

**线索逐渐收网：**
我仔细分析了主理人发来的 Nginx 日志和 Docker 进程情况。发现 Nginx 虽然返回了 `200 OK`，但流量根本不属于我们！
通过深入排查 Nginx 配置文件目录，我们抓出了“内鬼”：原来这台服务器上同时运行着另外一个名为 **openclaw** 的重型项目，而且它的 Nginx 配置文件**恰巧反常识地直接监听了宿主机的 `8080` 端口！**
而这，正好与我们前端 Docker 默认映射的 `8080` 发生了致命的**碰撞**！

**【一剑封喉】**：
两个“神仙打架”抢夺同一个门牌号，导致流量彻底迷乱。一旦看穿了病理，解药就非常简单了。我们立刻采取了“避战”策略：
1. 挥舞一行 `sed` 命令，瞬间将 Docker 容器对外的端口改迁到了无人涉足的 **`8081`**。
2. 将 Nginx 的 `odi.ai1717.cn` 反向代理流量精准制导至 `127.0.0.1:8081`。
3. `nginx -s reload` 重载！
